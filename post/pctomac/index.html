Switching from pc to mac

Sat, Jan 30, 2021 4-minute read

I’m a developer in the Microsoft ecosystem but I prefer Apple hardware, and since Microsoft introduced .NET 5 it is easy to work on your project using a mac.

This post shall describe you my experience regarding the switch from pc to mac, while the rest of the development team is working with a pc.

visual studio for mac

It works fine and is a very good IDE. A lot of details are much better than in the windows version - but I noticed the some points I want to share.

Shortcuts

Microsoft wants to give you the same “shortcut” experience like in windows – this makes things easy.

Out of the box, the 2 most important one’s work: “ctrl + .” and “ctrl + space” (these 2 I use the most)

What is a bit wired, is the fact that within visual studio for mac copy/paste/select-all is ctrl c/v/a while it is in the rest of the mac world cmd c/v/a.

CR / LF

Your “pc teammates” will recognize, that files you’ve created have different line endings, because when they open your files, they see this visual studio info.

Win uses: “\r\n”, also known as “CR / LF” or carriage return-linefeed.

Mac uses: “\r”, also known as “CR” or “Carriage return”.

Is this a problem? It is annoying – but I don’t see a problem here and I’m sure you can configure it in VS for mac somewhere…

Project Files

Sometimes VS for mac adds information to the .csproj file. For example, if you add a File to your project, VS for mac adds a whole new “Itemgroup” which is really not necessary – this is annoying when it comes to a Pull Request and you have to revert changes to the .csproj files

Git Support

The git support in VS for mac is ok (but nothing more).

But one thing really grinds my gears! If you switch the branch, let’s say from “develop” to “feature/xy” VS Studio for Mac still shows that you are working on development.

This is really annoying and hopefully fixed in one of the upcoming releases.

Therefore, I work with the git cli.

Extensions

… nothing to see here …

There are very few visual studio for mac extensions - but my beloved CodeMaid and R# are not available for mac.

Console / Terminal

I heavily use the console for git or dotnet commands – mostly just for fun (it feels so “hacky”). On mac, the console is a bit different.

Case sensitivity

It is cases sensitive by default. ‘DOTENET’ and ‘dotnet’ works the same, but ‘cd Projects’ and ‘cd projects’ are 2 different things.

Console start-up

What I loved the most in windows, you can navigate to every folder (using the file explorer) and just type in ‘cmd’ and the console starts within the folder. On mac, you have to click on a folder, and then select ‘show in terminal’…

the backslash

One of the most annoying things when switching from pc to mac is the backslash issue. Imagine you’re working a lot with cli commands, which includes folder structures, there is a difference between mac and pc.

Windows: cmd-something .\MyFolder\MyFile.txt

Mac OS: cmd-something ./MyFolder/MyFile.txt

I struggled with this a lot, but now I’m used to it.

Docker

Docker is the thing on a mac. It simply works – on windows I always struggled with my local docker instance but on mac - it just works.

And I’m using it much more than on a windows machine.

Sometimes I need a SQL server locally (I know it could be on Azure) – therefore I use docker.

Using a mac somehow forced me into docker (for running windows tools like SQL) that now, I use it for everything, everywhere. That’s a pretty cool side effect.

Conclusion

Yes, I would do it again! after 15 pc-years in professional software development I really enjoy the mac world.

That’s all for now.

happy coding.
